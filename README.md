### 1. **EditorConfig：前置配置（无需手动运行）**
它不是一个需要手动运行的命令行工具，而是**编辑器的配置文件**。
- 在你打开项目、编写代码时，编辑器会自动读取 `.editorconfig` 文件，统一缩进、编码、换行符等格式规则。
- 作用是从源头减少格式问题，属于“写代码时”就生效的前置步骤，不需要在命令行里执行。

---

### 2. **命令行工具的推荐运行顺序**
如果需要手动执行命令，或者在 CI/CD、pre-commit 钩子中配置，建议按以下顺序：

#### 第一步：`black`（自动格式化）
- 先让 `black` 自动修复所有格式问题（缩进、换行、空格、引号等）。
- 这样后面的检查工具就不会因为格式问题报错，避免重复检查和修改。

```bash
black your_project/
```

#### 第二步：`flake8`（快速静态检查）
- `flake8` 速度快，能快速发现语法错误、未定义变量、基础规范问题和代码复杂度问题。
- 先跑它可以及时修复明显的错误，避免带着基础问题去做深度检查。

```bash
flake8 your_project/
```

#### 第三步：`pylint`（深度质量检查）
- `pylint` 规则更细致，运行速度相对慢，放在最后可以在格式和基础问题都解决后，专注于代码质量、结构和逻辑合理性。
- 这样你可以更高效地处理深度问题，而不会被格式错误干扰。

```bash
pylint your_project/
```

---

### 为什么这个顺序更高效？
- 如果先跑 `flake8`/`pylint` 再跑 `black`，`black` 会修改代码格式，导致之前的检查结果失效，需要重新跑检查，浪费时间。
- 先格式化再检查，能确保检查工具只报告**真正需要手动修复的问题**，而不是格式问题。

---

### 进阶：用 pre-commit 自动化执行
在实际项目中，通常会用 `pre-commit` 钩子把这些工具串起来，在你提交代码时**自动按顺序执行**：
1. 触发 `black` 格式化
2. 运行 `flake8` 检查
3. 运行 `pylint` 检查
4. 只有所有检查通过，才允许提交代码

这样能确保每次提交的代码都符合所有规范，不需要手动记顺序。
